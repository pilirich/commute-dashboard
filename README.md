# 通勤儀表板 (Commute Dashboard)

## 📖 1. 使用目的
本網頁為一個客製化的「通勤決策儀表板」，旨在幫助使用者於每日上下班時段，根據最即時的交通動態（公車、YouBike、捷運及即時路況）來決定最佳的通勤方案。
核心亮點功能包含：
- **最晚出門時間排序**：系統會自動推算從家門口/公司走到站牌的步行時間，並換算出最晚必須出發的「死線 (Deadline)」，且總是將最急迫的方案排在最上方。
- **下一班車智慧推算**：當目前公車快進站（來不及走到站牌）時，系統會自動抓取全線資料，反推「再下一班車」的精確抵達時間並自動重新排序。

---

## ⚙️ 2. 基本設定與參數定義
所有的核心參數皆集中定義於 `bus.html` 內 `<script>` 區塊的最上方，未來若搬家或更換公司，僅需修改此區塊即可：

### API 金鑰設定 (Serverless Key Vault)
- **TDX Client ID / Secret** 與 **Google Maps API Key**：不再寫死於程式碼中。首次開啟網頁時將引導輸入，並安全綁定於瀏覽器 `localStorage` 內。

### 座標與站牌代碼
- `LOC_57_HOME`, `LOC_UBIKE_CO` 等: 各交通工具的實際搭乘/轉乘公開座標。
- `BUS_57_HOME`, `UBIKE_HOME` 等: 對應 TDX 系統中的站牌代碼 (StopID) 或租借站代碼 (StationID)。

### 方案定義 (Plans)
分為 `morning` (上班) 與 `afternoon` (下班) 兩個陣列，每個方案的參數包含：
- `gmMode`: Google Maps 計算模式 (`TRANSIT` 或 `BICYCLING`)。
- `isBus`: 標記是否為公車方案，以利過濾 Google Maps 步行的時間。
- `staticMins`: 在 Google API 失效或無資料時的預設「悲觀」車程總時間 (分鐘)。
- `extraMins`: (主要用於公車) 公車車程以外的固定消耗時間，包含騎 Ubike、走路、等捷運的總和。
- `preTransitMins`: 抵達「主要交通工具站牌」前所需的緩衝時間 (例如：從家裡走路到公車站需要 5 分鐘)。用於計算「最晚出門時間」。

## 🧠 3. 運作邏輯
1. **自動更新機制**：
   - 網頁載入後立即執行 `updateDashboard()`，隨後透過 `setInterval` **每 60 秒 (60000ms)** 自動更新一次。
2. **最晚出門時間 (Latest Departure Time)**：
   - 計算公式：`最晚出門時間 = 公車預估到站時間 (ETA) - preTransitMins`。
   - 若 `ETA < preTransitMins`，代表已經來不及，系統會觸發下一班車推算。
3. **動態時間緩衝與校正 (Time Calibration)**：
   為了彌補 API 預估與現實世界的落差，系統導入了嚴密的時空校正機制：
   - **公車 GPS 延遲預警 (-60秒)**：所有公車 ETA 在底層皆會強制扣除 60 秒，強迫提早出門，徹底消滅因 GPS 訊號落差而錯過車的風險。
   - **自行車路況校正 (+35%)**：因 Google Maps 自行車模式並未考量台灣高密度的紅綠燈與 Ubike 較重的車身，所有 API 回傳的騎乘時間皆會乘上 **1.35 倍**。
   - **固定辦公室進出成本 (+5分鐘)**：所有路線皆加入 5 分鐘的固定緩衝，用以吸收「等電梯上下樓」與「從辦公桌走到公司大門口」的隱形步行時間。
4. **「再下一班車」智慧推算模型 (Next Next Bus Heuristic)**：
   - 當判定第一班車來不及時，`fetchBus` 會自動調閱該路線「所有的站牌動態 (ETA)」與「站牌順序 (DisplayStopOfRoute)」。
   - 由目標站牌往回追溯，當發現預估時間出現 **> 180 秒的時間斷層 (Jump)** 時，即代表抓到了「後面那一班車」的位置。
   - 系統會將該斷層時間加上第一班車的剩餘時間，得出極度精確的下一班車 ETA。
5. **靜態快取機制 (Caching)**：
   - **站牌順序 (`seqCache`)**：存於瀏覽器 `localStorage`，設定 **7 天過期** (`604800000` ms)。切換上下班或重新整理時不須重新向 TDX 請求靜態資料。
   - **TDX Token**：存於記憶體變數，以 `expires_in - 60s` 為有效期限，避免每分鐘重複請求驗證。
6. **Ubike 目的地車柱防呆**：
   - 不僅查詢借車點的車輛數，系統會同步檢查「還車點」的空柱數。當終點空柱過少 (≤2) 或全滿 (0) 時，會主動發出黃色預警或紅色滿站警告，避免騎到目的地無法還車的窘境。
7. **Ubike 備用站智慧降級 (Smart Fallback)**：
   - 當首選借車站「無車可借」，或首選還車站「滿站」時，系統會自動導向鄰近的備用站（例如：福和國中、永和自由街、板橋第一運動場）。
   - 發生降級時，畫面上會動態插入橘色警告（如：`⚠️ 改至福和國中借車 (+3分)`），並且**自動將多出的步行時間加入死線計算**，確保抵達時間依舊精準無誤。
8. **零延遲切換 (Zero-Latency Toggle) 與 API 終極打包**：
   - 每次背景自動更新時，程式會利用 TDX 支援全線回傳與 `$filter` 批次查詢的特性，將「上下班」所需的雙向公車全線資料、以及所有借還車 Ubike 站點 **一次抓齊並快取於記憶體**。
   - 切換上午/下午模式時，TDX 交通資料可瞬間重新渲染，僅觸發 Google Maps 的單向路況計算 (Lazy Load)，實現畫面幾乎無延遲的無縫切換。

## 🔌 4. API 規格
本系統高度依賴以下兩大平台的 API：

### Google Maps Platform
- **Directions API**: 使用 `transitOptions` 指定出發時間（早上 8:00 或 晚上 18:00 或當下時間）以獲取包含即時塞車路況的公車搭乘時間，並自動過濾掉純粹步行的時間 (`foundTransit` 邏輯)。

### TDX 運輸資料流通服務平臺
- **Auth Token** (`/auth/realms/TDXConnect/protocol/openid-connect/token`): 使用 `client_credentials` 模式取得 Bearer Token。
- **Bus ETA** (`/api/basic/v2/Bus/EstimatedTimeOfArrival/City/{City}/{RouteName}`): 取得全線預估到站時間。
- **Bus Route Sequence** (`/api/basic/v2/Bus/DisplayStopOfRoute/City/...`): 取得路線站牌順序 (用於推算下一班車)。
- **Bike Availability** (`/api/basic/v2/Bike/Availability/City/...`): 取得 YouBike 2.0 即時可借車輛數。

## 🚦 5. API 呼叫限制與額度評估

### TDX API (基礎會員 - 免費方案)
- **官方限制**：每分鐘 5 次 (5 req/min/key)。
- **本系統消耗**：每分鐘自動更新 1 次，每次更新最高消耗 **3 次** API 請求 (公車全線 ETA * 2 + Ubike 雙向站點打包查詢 * 1)。*(註：Token 與站牌順序因有快取保護，不列入常態消耗)*
- **安全評估**：恰好控制在免費額度的 5 次以內。透過巧妙的 API 批次打包技術，我們在不增加任何請求次數的前提下，成功快取了雙向交通資料。

### Google Maps Directions API
- **官方限制**：Google Maps Platform 已於 2025 年 3 月 1 日實施全新計費模式，Directions API 被歸類為「Essentials」服務，每個帳號每月享有 **10,000 次**的獨立免費額度。額度於每個月第一天重新計算。
- **本系統消耗**：採「懶人載入(Lazy Load)」，每次背景更新只會呼叫 3 次 API (僅針對當前啟用的上下班模式)。若每天上下班各開啟 5 分鐘 (每月 22 天工作日)，每月總消耗約為 **1,452 次**。
- **安全評估**：佔免費額度的 14.5%，遠低於一萬次上限，可絕對安心常駐開啟。

## 🔐 6. 資安與金鑰管理 (Serverless Key Vault)
為了讓此專案能百分之百安全地開源並託管於 GitHub Pages 等公開平台，系統實作了**「無伺服器金鑰金庫」**架構：
1. **零明文原始碼**：`index.html` 內完全不包含任何 Google Maps 或 TDX 的 API Key，從根本上杜絕金鑰外洩風險。
2. **本機綁定儲存**：在新設備初次開啟網頁時，系統會顯示初始設定介面，要求輸入金鑰。輸入後，金鑰將直接綁定並加密儲存於該瀏覽器的 `localStorage` 中，絕對不會上傳至任何網路。
3. **無縫動態載入**：後續每次開啟網頁，系統會默默從 `localStorage` 讀取金鑰，動態注入 Google Maps 腳本並獲取 TDX Token，實現無需重複登入的無縫體驗。
4. **支援密碼管理員**：設定介面支援 Chrome 密碼管理員自動填寫 (Autofill)，強烈建議您將金鑰儲存於您的 Google 帳號密碼庫中，換設備時即可一鍵登入。
5. **隱私座標隔離 (Privacy Geolocation Isolation)**：原始碼中僅保留公車站牌、捷運站、Ubike站等「公開大眾運輸節點」的經緯度。您的住家門口與辦公室的精確私人座標已被徹底移除，步行時間改以常數（固定分鐘數）獨立計算，徹底防止個人住址外洩。

## 🛠️ 7. 未來維護與修改須知
1. **API Key 過期或失效**：
   - 若儀表板出現 `TDX Auth failed`，請至 TDX 官網重新申請/確認 `CLIENT_ID` 與 `CLIENT_SECRET`。
   - 若底部顯示 `⚠ Google API 尚未連線`，且預估時間變回固定值 (如 55 分鐘)，請至 Google Cloud Console 確認 API Key 是否有效，以及 `Directions API` 服務是否已啟用並綁定帳單。
2. **新增公車路線**：
   - 若未來需新增如「綠2」等公車，請於 `getBusDataObj` 呼叫時帶入適當的 `headwayMins`（班距預設值，目前 57 為 15分，897 依上下班時段動態分為 30/50分）。當「動態下一班車推算」因資料不足失效時，系統會以此班距作為容錯退路。
3. **捷運 API (未來擴充)**：
   - 目前捷運視為「隨時可搭 (3分鐘班距)」，未串接動態資料。若未來搬家導致需搭乘班距較長的台鐵或捷運末端路線，可參考 TDX Rail API 進行動態串接。
4. **localStorage 除錯**：
   - 若因公車改線導致到站時間計算異常，可於瀏覽器按 `F12` 進入 Console 執行 `localStorage.removeItem('tdx_seq_cache')` 強制清空站牌快取。
